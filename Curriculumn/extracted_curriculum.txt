Full module structure, learner journey, outcomes & SETA alignment
2.1 Course Overview
Cloud Launchpad is the gateway course of the Cloud Computing track at Ginashe Digital Academy. It is designed for learners with no prior technical background — school leavers, career changers, and early-workforce individuals — and equips them with the foundational knowledge and practical skills to pursue entry-level cloud roles or continue to the Associate level of the track.

On successful completion of Cloud Launchpad, learners will be able to:
Explain cloud computing concepts, service models, and deployment types
Administer a Linux server using the command line interface

Demonstrate POPIA awareness in the context of cloud data handling
Estimate and interpret a basic cloud cost forecast
Present a cloud solution architecture to a mixed technical and non-technical audience

4.1 Module Overview
Linux is the operating system that powers the cloud. More than 90% of public cloud workloads run on Linux, all major cloud providers offer Linux-based compute instances as their primary offering, and every cloud administrator, DevOps engineer, and solutions architect must be comfortable with the Linux command line. This module transforms learners from cloud consumers to cloud operators — by the end of Week 4, every learner will have a running web server they configured entirely from the command line.

4.2 Lab Environment Setup
All CLI labs in this module require a live Linux environment. The recommended setup is an Ubuntu 22.04 LTS EC2 t2.micro instance (free tier eligible) per learner provisioned in the AWS af-south-1 (Cape Town) region. The following must be confirmed before Lesson 1 begins:
Each learner has an active AWS Free Tier account (set up in Module 1)
EC2 instance is running Ubuntu 22.04 LTS in af-south-1
Security group allows inbound SSH (port 22) from the class IP range and HTTP (port 80) from anywhere
Learners can connect via SSH using PuTTY (Windows) or terminal (Mac/Linux)
Fallback: browser terminal at https://bellard.org/jslinux is tested and working
A second shared EC2 instance is available for SSH/SCP exercises in Lesson 4
4.3 Module Learning Outcomes
On completion of Module 2 learners will be able to:
Explain the history and philosophy of Linux and its role in cloud infrastructure
Navigate the Linux file system hierarchy using the command line
Create, move, copy, and delete files and directories using core shell commands
Interpret and modify file permissions and ownership using chmod, chown, and octal notation
Manage software packages using apt (Debian/Ubuntu) and describe yum/dnf for RHEL
Use text editors (nano and vi) to create and edit configuration files
Write and execute a basic Bash shell script to automate a repetitive task
Monitor system processes, memory, and disk using ps, top, df, and free
Connect to a remote Linux server via SSH and transfer files using SCP

Networking is the connective tissue of the cloud. DynamoDB and NoSQL fundamentals (40 min) — Relational databases (RDS) enforce a rigid schema with tables, rows, columns, and foreign key relationships — excellent for structured transactional data. 0:00
Discussion
The Capital One cloud breach — IAM misconfiguration (20 min) — In 2019 a misconfigured IAM role on an EC2 instance allowed an attacker to query the AWS instance metadata service and obtain temporary credentials with excessive privileges. The attacker exfiltrated over 100 million customer records. Capital One paid an $80 million regulatory fine and faced significant reputational damage. The root cause was a single IAM role with broader permissions than necessary. This sets the tone: IAM is not optional configuration — it is the foundation of all cloud security.
0:20
Lecture
IAM entities and the AWS permission model (50 min) — Root account: omnipotent access to everything in the account; must never be used for daily work; must be locked down with MFA immediately after account creation. IAM users: human identities with long-term username, password, and optional access keys. IAM groups: attach policies to groups and add users to groups — never attach policies directly to individual users at scale. IAM roles: temporary credential providers assumed by services, EC2 instances, Lambda functions, or federated users — no long-term access keys, credentials auto-rotate. Policies: JSON documents defining Allow or Deny actions on specific resources. Policy types: AWS managed policies (maintained by AWS, updated automatically), customer managed policies (you write and maintain), and inline policies (embedded directly in one entity, not reusable). Policy evaluation logic: explicit Deny always overrides any Allow — even if multiple Allow policies apply.
1:10
Lecture
IAM policy JSON anatomy (30 min) — Structure: Version (always 2012-10-17), Statement (array of permission blocks), Effect (Allow or Deny), Principal (who the policy applies to — not used in identity policies), Action (list of API actions such as s3:GetObject or ec2:DescribeInstances), Resource (the ARN of the resource the action applies to), and Condition (optional additional constraints such as IP range or MFA requirement). Wildcard actions using asterisk — when acceptable (read-only wildcard on a tightly scoped resource) and when dangerous (Action: * on Resource: *). ARN format: arn:aws:s3:::bucket-name/* for all objects in a specific bucket. The principle of least privilege in practice: start with no permissions and add only what is explicitly needed for the specific task.

IAM roles for EC2 and the service account pattern (35 min) — The fundamental problem with access keys on EC2 instances: if an attacker gains SSH access, they find the keys in ~/.aws/credentials and can use them from anywhere in the world indefinitely. 1

2

The AWS Shared Responsibility Model (45 min) — AWS is responsible FOR the cloud: physical data centres including the Cape Town af-south-1 facility, networking hardware, hypervisor software, and managed service software such as RDS engine patches. Lecture
Encryption fundamentals and AWS KMS (40 min) — Symmetric encryption: the same key both encrypts and decrypts (AES-256 — the standard for data at rest). Asymmetric encryption: a public key encrypts, only the corresponding private key decrypts (RSA — used in TLS handshakes and digital signatures). Encryption at rest protects data when it is stored on a disk or in an object store from physical media theft or unauthorised storage-level access. Encryption in transit protects data as it travels across a network from eavesdropping and man-in-the-middle attacks (TLS 1.2 and 1.3). AWS Key Management Service: a managed service for creating and controlling encryption keys. Customer Master Keys (CMKs): either AWS-managed (free, managed by AWS) or customer-managed ($1/month, full key policy control). KMS integrates natively with S3, EBS, RDS, Secrets Manager, CloudTrail, and Lambda.

Encryption in transit — HTTPS with ACM and Application Load Balancer (35 min) — Why running HTTP is dangerous for any application that handles user credentials, session tokens, or personal information: all data travels in plaintext and can be intercepted by anyone with network access between the client and server. The 2020 SolarWinds supply chain attack (20 min) — Attackers compromised SolarWinds' software build pipeline and inserted malicious code into a legitimate signed software update package for the Orion platform. Cloud incident response — 6 phases in depth (45 min) — Phase 1 Prepare: develop and test IR runbooks before an incident occurs; run AWS GameDays to simulate breach scenarios; create a pre-authorised forensic IAM role with read-only access to all services for use during investigations; maintain an isolated forensic VPC with no internet access for safe analysis; ensure CloudTrail, GuardDuty, and VPC Flow Logs are pre-enabled on all accounts; establish an out-of-band communications plan that does not rely on potentially compromised systems. Phase 2 Identify: detect anomalies via GuardDuty machine learning alerts, CloudTrail unusual activity patterns, VPC Flow Log traffic spikes, billing alerts triggered by unexpected resource provisioning such as cryptomining EC2 instances. Phase 3 Contain: immediately isolate a compromised EC2 instance by replacing its security group with a deny-all rule; revoke or deactivate the associated IAM role or user; take an EBS snapshot of the instance for forensic preservation before any changes; rotate all credentials that may have been exposed. Phase 4 Eradicate: terminate the compromised instance after forensic snapshot; delete malicious IAM users, access keys, or policies created by the attacker; patch or remediate the specific vulnerability that was exploited. Phase 5 Recover: launch a clean replacement instance from a known-good AMI; restore application data from the last verified clean backup; validate integrity by comparing checksums; monitor intensively for recurrence. Phase 6 Lessons Learned: conduct a blameless post-mortem within 5 business days; identify the root cause, contributing factors, and detection gaps; update IR runbooks with findings; implement preventive technical controls to prevent recurrence.

8.6 Four-Phase Project Plan
The capstone runs over two weeks and is structured into four distinct phases. Each phase has defined deliverables that must be completed before moving to the next.
Phase 1 — Briefing and Architecture Design (Week 11, Day 1 — 5 hours)
Phase 1 objectives
Select or receive a capstone scenario assignment
Understand the client brief and extract key technical requirements
Design a cloud architecture on paper before touching the console
Submit the architecture design for facilitator sign-off
Produce an initial cost estimate range using the AWS Pricing Calculator
Facilitated session plan — 5 hours
Hours 0–1: Facilitator presents all 6 scenarios with a 10-minute brief on each, highlighting the key technical challenge and POPIA considerations. Learners indicate their first and second scenario preferences. Facilitator assigns scenarios aiming for distribution across the class (no more than 3 learners per scenario in a class of 15).
Hours 1–3: Independent design time. Each learner draws their architecture on a blank A3 sheet provided. The design must show: VPC boundary, public and private subnets, all AWS services used, security group boundaries, data flow arrows with encryption annotations, and IAM role assignments. Facilitator circulates, asking probing questions: 'Why did you choose RDS over DynamoDB here?' 'Where is your encryption at rest?' 'What happens if this EC2 instance fails?'
Hours 3–4: Peer review pairs. Each learner presents their design to a partner for 10 minutes. Partner reviews against a 10-point design checklist (VPC correct, subnets correct, encryption present, IAM roles used, cost-conscious choices, POPIA-relevant controls). Feedback recorded in writing.
Hours 4–5: Facilitator sign-off. Each learner has a 5-minute conversation with the facilitator about their design. Facilitator approves or requires specific changes before the learner may begin building. No learner starts Phase 2 without a signed-off design.
Phase 1 deliverable: hand-drawn and facilitator-signed architecture diagram, submitted as a PoE artefact.
Phase 2 — Build and Deploy (Week 11, Days 2–3 — 7 hours)
Phase 2 objectives
Deploy the signed-off architecture on AWS using the console and CLI
Configure all security controls from the design
Verify all services are running and the application is operational
Troubleshoot any deployment issues with facilitator support
Document all deployment steps with screenshots as PoE evidence
Build sprint structure — 7 hours over 2 days
This is the most intensive phase. Learners work independently with facilitated support. The facilitator does not solve problems for learners — they ask questions that guide learners to the solution themselves ('What does the error message say? Where would you look in the AWS documentation?'). This develops problem-solving skills that are essential in real cloud roles.
Day 1 (4 hours): VPC and networking setup, EC2 instance launch and configuration, application deployment. At the 2-hour mark the facilitator does a structured check-in: is the VPC up? Is EC2 running? Is SSH working? Learners who are stuck on the same issue for more than 30 minutes are paired with a peer who has resolved that issue.
Day 2 (3 hours): Database deployment and connection, S3 configuration, IAM role setup, security hardening, and end-to-end functionality testing. At the 2-hour mark: structured check-in. Does the application read from and write to the database? Are security groups restricted? Is encryption enabled? Remaining time: polishing and capturing screenshots.

Phase 3 objectives

Documentation session structure — 4 hours
Hour 1: Architecture diagram finalisation. Learners refine their hand-drawn diagram or reproduce it digitally using a tool of their choice (draw.io, Lucidchart, PowerPoint, or pen and paper are all acceptable). All components must be labelled with service names and security annotations. Data flow arrows must indicate encryption status.
Hours 1–2: Security hardening report. Using the Module 5 template, learners document: (1) current security controls implemented in the capstone, (2) POPIA compliance mapping for all 8 conditions, (3) findings table with at least 3 identified gaps and their recommended remediations, (4) Well-Architected Security Pillar assessment score.
Hour 2–3: Cost estimate. Using the AWS Pricing Calculator, learners build a monthly cost estimate for their deployed architecture. The estimate must be converted to South African Rand using the current exchange rate (the facilitator provides the day's USD/ZAR rate). At least two cost optimisation recommendations must be included with their projected savings.
Hour 3–4: Reflective journal and PoE compilation. The reflective journal asks learners to answer: What did I learn in Cloud Launchpad that surprised me most? What was the hardest technical concept? What cloud role do I now want to pursue and why? How would I approach this project differently with more time? PoE artefacts from all 6 modules are organised into a logical submission.
Phase 3 deliverable: complete PoE submission including architecture diagram, security report, cost estimate, and reflective journal.

Presentation day schedule
Learners present in groups of 4–5 per session. Each presentation is 15 minutes followed by 5 minutes of panel Q&A. The panel consists of the facilitator and one guest — ideally a practising cloud professional sourced from GDA's industry network (an alumnus, a Ginashe Digital client, or an MICT SETA industry partner).
Required presentation structure — 15 minutes
Slide 1 (1 min): Introduction — learner name, scenario name, the business problem in one sentence
Slide 2–3 (2 min): Client background — who is the client, what is the problem, why cloud is the right solution
Slide 4–5 (3 min): Architecture — the diagram, each service and why it was chosen, data flow explanation
Slide 6 (2 min): Security and POPIA — encryption, IAM, network security, POPIA conditions met
Slide 7 (2 min): Cost estimate — monthly cost in ZAR, pricing model choices, top optimisation recommendation
Slide 8 (2 min): Live demo or screen recording — show the running application
Slide 9 (3 min): Reflections and recommendations — what you would do next with more time or budget
